# JifeLog 회원가입 페이지 PRD

## 문서 목적
- 현재 [components/pages/SignupPage.tsx](/Users/je/vscode/jifelog-project/components/pages/SignupPage.tsx)는 모든 입력 항목을 한 화면에 동시에 노출하는 단일 폼 구조다.
- 이 문서는 기존 입력 컴포넌트와 입력 항목은 유지하면서, 2026년 기준 가입 UX 트렌드에 맞는 단계적 노출 구조로 회원가입 화면을 재정의하기 위한 기획서다.
- 목적은 입력 부담을 줄이고, 첫 진입 인상과 완료율을 함께 높이는 것이다.

## 현재 구조 요약
- 이름, 이메일, 인증요청, 인증번호, 비밀번호, 비밀번호 확인, 약관 동의, 제출 버튼이 한 카드 안에 모두 배치되어 있다.
- 이메일 인증 상태에 따라 일부 필드가 조건부로 열리지만, 전체 구조는 여전히 "긴 폼"으로 인식된다.
- 가입 화면의 분위기는 어둡고 감성적이지만, 입력 흐름 자체는 최신 가입 패턴에 비해 정적이다.

## 문제 정의
- 첫 진입 시 입력해야 할 항목이 많아 보여 심리적 진입 장벽이 높다.
- 사용자에게 "다음에 무엇을 입력하면 되는지"가 자연스럽게 안내되지 않는다.
- 이메일 인증 이후에도 비밀번호, 약관 동의까지 한 번에 보여 정보 밀도가 높다.
- 상단 브랜딩과 하단 입력 폼이 분리되어 있지 않아 화면 구조의 긴장감이 약하다.

## 리디자인 목표
1. 처음에는 가장 쉬운 1개 입력만 보여 주고, 완료에 따라 다음 입력이 열리는 구조로 바꾼다.
2. 기존 입력 필드와 데이터 요구사항은 유지하되, 화면 인식은 "긴 폼"이 아니라 "짧은 단계"처럼 느껴지게 만든다.
3. 사용자가 지금 어느 단계인지, 다음에 무엇을 해야 하는지 명확히 이해할 수 있게 한다.
4. 모바일 기준으로 시선이 흔들리지 않는 집중형 가입 화면을 만든다.

## 유지 조건
- 유지 대상 입력 항목
  - 이름
  - 이메일
  - 이메일 인증번호
  - 비밀번호
  - 비밀번호 확인
  - 약관 동의
- 기존 입력 컴포넌트 자체는 최대한 유지한다.
- 이메일 유효성 검사, 인증 상태, 비밀번호 일치 확인 같은 현재 로직 방향은 유지한다.

## 핵심 전략
### UX 전략 한 줄
한 번에 다 보여주지 말고, 입력이 완료될수록 다음 단계가 열리는 progressive disclosure 구조를 사용한다.

### 구조 전략
- 한 화면 전체를 여러 페이지로 나누는 완전한 멀티스텝 폼이 아니라,
- "한 카드 안에서 아래 단계가 순차적으로 열리는 단일 페이지형 단계 구조"를 채택한다.

이 방식은 구현 복잡도를 낮추면서도 최신 가입 UX 인상을 만들기에 적합하다.

## 트렌드 반영 방향
- 첫 화면에서 사용자가 보는 입력은 1개 또는 2개를 넘기지 않는다.
- 이전 단계가 유효해지면 다음 필드가 자연스럽게 reveal 된다.
- 이미 완료된 단계는 축약 상태 혹은 완료 상태로 인지되도록 표시한다.
- 단계 자체는 복잡한 스텝퍼보다 "흐름이 이어진다"는 느낌으로 보여 준다.
- CTA는 모든 입력을 마친 뒤에만 강하게 드러난다.

## Visual Thesis
어두운 배경 위에 하나의 세련된 입력 컬럼만 떠 있는 집중형 가입 화면. 정보량은 줄이고, 입력이 아래로 조용히 열리며 신뢰감 있게 완성된다.

## Content Plan
1. Intro: 브랜드와 가입 이유를 짧게 전달
2. Step Input Flow: 이름 > 이메일 > 인증 > 비밀번호 > 약관
3. Completion CTA: 가입 완료 버튼과 로그인 링크

## Interaction Thesis
- 첫 진입 시 카드 전체가 아니라 첫 입력 블록만 강하게 주목된다.
- 유효한 입력이 끝나면 다음 블록이 아래에서 부드럽게 등장한다.
- 완료된 항목은 체크 상태와 함께 안정적으로 접히거나 유지되며, 사용자는 자신의 진행 상태를 잃지 않는다.

## 타겟 사용자
- 모바일에서 빠르게 가입을 마치고 싶은 신규 사용자
- 감성적인 제품 화면을 선호하지만 입력 과정은 단순하길 원하는 사용자
- 긴 가입 폼에 피로감을 느끼는 사용자

## 화면 구조 재정의
## 1. 상단 영역
### 목적
- 서비스 정체성과 가입 의도를 짧게 전달

### 요구사항
- 현재처럼 `회원가입`만 쓰기보다, 브랜드 맥락이 드러나는 짧은 한 줄을 보강한다.
- 예: "당신의 하루를 연결할 준비를 시작하세요"
- 상단 카피는 2줄 이내로 제한한다.
- 뒤로가기 액션은 유지하되 입력 집중을 해치지 않게 조용한 톤으로 둔다.

## 2. 입력 카드 구조
### 목적
- 폼을 하나의 긴 덩어리가 아니라 단계적으로 열리는 흐름으로 보이게 만든다.

### 구조 원칙
- 카드 안에는 현재 단계와 다음 단계만 강하게 보인다.
- 아직 도달하지 않은 단계는 숨기거나 약하게 대기 상태로 둔다.
- 완료된 단계는 체크 아이콘과 함께 완료 상태가 표시된다.

### 권장 단계 순서
1. 이름 입력
2. 이메일 입력
3. 이메일 인증번호 입력
4. 비밀번호 입력
5. 비밀번호 확인
6. 약관 동의
7. 회원가입 CTA 활성화

## 3. 단계별 노출 규칙
### Step 1. 이름
- 첫 진입 시 가장 먼저 보이는 입력
- 이름이 비어 있지 않으면 다음 이메일 단계가 열린다.
- 별도 검증은 강하지 않게 두되, 공백만 입력된 경우는 다음 단계 열림 대상에서 제외 가능

### Step 2. 이메일
- 이름 입력 이후 노출
- 이메일 형식이 유효할 때만 인증 요청 버튼을 활성화
- `인증요청` 버튼은 이 단계의 주 액션

### Step 3. 이메일 인증
- 인증 요청 이후에만 노출
- 인증 전에는 이후 비밀번호 단계가 보이지 않도록 한다.
- 인증이 완료되면 이메일 영역은 잠금 상태 또는 완료 상태로 보일 수 있다.

### Step 4. 비밀번호
- 이메일 인증 완료 후 노출
- 비밀번호 규칙은 짧게 보조 문구로 제공
- 예: `8자 이상 입력해주세요`

### Step 5. 비밀번호 확인
- 비밀번호 입력이 시작되면 노출
- 불일치 시 즉시 오류 표시
- 일치 시 시각적으로 안도감을 주는 체크 상태를 유지

### Step 6. 약관 동의
- 비밀번호 확인까지 끝난 뒤 노출
- 가입 직전 마지막 확인 단계 역할
- 링크는 유지하되 약관 블록이 지나치게 커 보이지 않게 압축한다.

### Step 7. 가입 버튼
- 모든 선행 조건 충족 후 강하게 활성화
- 비활성 상태에서도 버튼 영역은 유지하되, 시각적 강조는 최소화

## UI 구성 원칙
- 한 번에 2개 이상의 미완료 입력이 강하게 경쟁하지 않게 한다.
- 각 단계는 "섹션"이 아니라 하나의 입력 블록처럼 보여야 한다.
- 카드형 박스를 반복해서 쌓지 않는다.
- 구분선, 여백, 상태 아이콘만으로 단계 차이를 만든다.

## 상태 설계
### 기본 상태
- 현재 입력 대기 중

### 진행 상태
- 이전 단계 완료, 현재 단계 활성

### 완료 상태
- 체크 아이콘 + 값 유지 또는 축약 표시

### 오류 상태
- 이메일 형식 오류
- 비밀번호 불일치
- 미인증 상태
- 약관 미동의 상태

## 카피 가이드
- 짧고 안내적인 문장 사용
- "회원가입을 완료하세요"보다 "다음 정보를 입력해주세요"처럼 단계 안내가 읽히게 작성
- 에러 문구는 차갑지 않고 명확하게
- 예:
  - `올바른 이메일 형식을 입력해주세요`
  - `비밀번호가 일치하지 않습니다`
  - `이메일 인증을 완료해주세요`

## 비주얼 가이드
- 현재 다크 배경과 teal 포인트는 유지
- 전체 화면은 여전히 프리미엄하고 차분해야 함
- 입력 카드 하나만 중심에 두고, 장식 요소는 지금보다 더 절제
- 배경 효과는 유지 가능하나 입력 영역 집중도를 해치지 않아야 함
- 모바일에서는 카드 폭을 더 넓게 체감시키고, 좌우 여백보다 입력 집중을 우선

## 모션 가이드
- 단계가 열릴 때 아래에서 위로 짧게 등장
- 완료된 항목은 과한 애니메이션 없이 상태 변화만 전달
- 오류는 흔들림보다 색상과 짧은 문구로 전달
- 전체 페이지에서는 2~3개의 반복 가능한 모션 패턴만 사용

## 권장 레이아웃 패턴
### 옵션 A. 단일 카드 내 순차 reveal
- 가장 추천하는 구조
- 현재 컴포넌트 재사용성이 높고 구현 부담이 낮다.

### 옵션 B. 상단 진행 라벨 + 하단 단일 입력
- 단계감은 강하지만 현재 코드 구조 수정 범위가 커질 수 있다.

이번 요구사항에서는 옵션 A를 기본안으로 채택한다.
