# JifeLog 회원가입 페이지 PRD

## 문서 목적
- 현재 [components/pages/SignupPage.tsx](/Users/je/vscode/jifelog-project/components/pages/SignupPage.tsx)는 모든 입력 항목이 한 카드 안에 모여 있는 단일 폼 구조다.
- 이 문서는 기존 입력 항목과 검증 로직은 유지하면서, 2026년 기준 가입 UX 트렌드에 맞는 `순차 노출형 회원가입 화면`으로 재정의하기 위한 기획서다.
- 핵심 목표는 입력 부담을 줄이고, 모바일에서도 좁아 보이지 않는 화면 구조로 가입 완료율을 높이는 것이다.

## 현재 구조 요약
- 이름, 이메일, 인증요청, 인증번호, 비밀번호, 비밀번호 확인, 약관 동의, 제출 버튼이 한 화면에 가깝게 배치되어 있다.
- 일부 입력은 조건부로 열리지만 전체 인상은 여전히 긴 카드형 폼이다.
- 입력 영역이 카드 안에 갇혀 있어 모바일에서 폭이 더 좁게 체감된다.

## 문제 정의
- 첫 진입 시 입력해야 할 정보가 많아 보여 심리적 부담이 크다.
- 사용자는 가입 흐름보다 폼 덩어리를 먼저 인식하게 된다.
- 카드 내부에 입력이 반복되면서 화면이 조밀하고 답답해 보인다.
- 모바일에서는 여백 대비 입력 폭이 줄어들어 더 좁고 답답한 인상을 준다.

## 리디자인 목표
1. 사용자는 한 번에 하나의 입력 흐름만 보게 한다.
2. 이전 입력이 유효해지면 다음 입력이 자연스럽게 열린다.
3. 진행 상황을 별도 컴포넌트로 설명하지 않고, 사용자는 지금 보이는 입력과 다음 나타나는 입력만 따라가면 되게 만든다.
4. 카드형 박스가 아닌 화면 일체형 레이아웃으로 바꿔 모바일 체감 폭을 넓힌다.

## 유지 조건
- 유지 대상 입력 항목
  - 이름
  - 이메일
  - 이메일 인증번호
  - 비밀번호
  - 비밀번호 확인
  - 약관 동의
- 기존 입력 로직 방향 유지
  - 이메일 형식 검사
  - 인증 요청 후 인증번호 입력 노출
  - 비밀번호 일치 확인
  - 약관 동의 후 가입 가능
- 입력 필드 자체는 최대한 재사용한다.

## 핵심 전략
### UX 전략 한 줄
설명하지 말고, 다음 입력이 열리게 하라.

### 구조 전략
- 완전한 멀티페이지 가입이 아니라 한 화면 안에서 입력 블록이 아래로 순차적으로 열리는 방식으로 간다.
- 사용자는 별도 스텝 표시, 진행률, 요약 상태를 볼 필요가 없다.
- 지금 입력해야 할 것과 바로 다음에 열리는 입력만 보이면 된다.

## 트렌드 반영 방향
- 첫 화면에서는 가장 쉬운 입력 1개만 강하게 보인다.
- 이전 입력이 완료되면 다음 입력이 아래에서 짧게 등장한다.
- 완료 상태를 별도의 진행 UI로 과장하지 않는다.
- 입력 흐름은 조용하고 자연스럽게 이어지며, 화면 자체가 하나의 리듬으로 느껴져야 한다.
- CTA는 마지막 단계에서만 강하게 드러난다.

## Visual Thesis
다크 톤의 배경 위로 입력 영역이 화면에 자연스럽게 스며든 가입 화면. 박스 안의 폼이 아니라 화면 자체가 입력 흐름이 되며, 모바일에서도 넓고 시원하게 느껴진다.

## Content Plan
1. Intro: 브랜드와 가입 이유를 짧게 전달
2. Sequential Input Flow: 이름 > 이메일 > 인증 > 비밀번호 > 약관
3. Final Action: 회원가입 CTA와 로그인 링크

## Interaction Thesis
- 첫 입력은 이미 화면 일부처럼 놓여 있어 진입이 가볍다.
- 입력이 유효해질수록 다음 블록이 아래에서 부드럽게 이어진다.
- 사용자는 별도 상태판 없이도 흐름을 잃지 않는다.

## 타겟 사용자
- 모바일에서 빠르게 가입을 마치고 싶은 신규 사용자
- 긴 폼보다 짧게 이어지는 입력 흐름을 선호하는 사용자
- 감성적인 제품 화면은 좋아하지만, 가입 과정은 단순하길 원하는 사용자

## 화면 구조 재정의
## 1. 상단 영역
### 목적
- 서비스 정체성과 가입 맥락을 짧게 전달

### 요구사항
- `회원가입`만 노출하지 말고 제품 맥락이 읽히는 짧은 문장을 함께 둔다.
- 예: `당신의 하루를 연결할 준비를 시작하세요`
- 상단 카피는 2줄 이내
- 뒤로가기 액션은 유지하되 화면 몰입을 해치지 않게 작고 조용하게 둔다.

## 2. 화면 본문 구조
### 목적
- 입력이 카드 안에 갇히지 않고 화면 전체 흐름 안에서 자연스럽게 이어지게 만든다.

### 구조 원칙
- 큰 카드 하나를 중앙에 띄우는 구조를 쓰지 않는다.
- 배경, 카피, 입력 영역이 하나의 레이어처럼 이어진다.
- 입력 영역은 화면 중앙 혹은 한쪽 축에 정렬하되, 독립 카드처럼 보이지 않아야 한다.
- 모바일에서는 좌우 여백을 줄이고 입력 폭을 최대한 넓게 체감시킨다.

## 3. 입력 노출 구조
### 목적
- 사용자가 현재 입력에만 집중하고, 다음 입력은 조건이 만족될 때만 자연스럽게 보이도록 한다.

### 권장 순서
1. 이름 입력
2. 이메일 입력
3. 이메일 인증번호 입력
4. 비밀번호 입력
5. 비밀번호 확인
6. 약관 동의
7. 회원가입 버튼 활성화

## 단계별 노출 규칙
### Step 1. 이름
- 첫 진입 시 유일하게 강하게 보이는 입력
- 이름이 비어 있지 않으면 이메일 입력이 열린다.

### Step 2. 이메일
- 이름 입력 이후 노출
- 이메일 형식이 유효할 때만 `인증요청` 버튼 활성화
- 이 단계까지도 화면은 간결해야 한다.

### Step 3. 이메일 인증
- 인증 요청 이후에만 노출
- 인증 완료 전에는 비밀번호 단계가 보이지 않는다.
- 사용자는 이 블록만 처리하면 다음 입력이 열린다는 점을 직관적으로 이해해야 한다.

### Step 4. 비밀번호
- 인증 완료 후 노출
- 보조 문구는 짧게
- 예: `8자 이상 입력해주세요`

### Step 5. 비밀번호 확인
- 비밀번호 입력 시작 후 노출
- 불일치 시 즉시 오류 표시
- 과한 상태 장식 없이 입력과 문구만으로 충분히 이해되게 한다.

### Step 6. 약관 동의
- 비밀번호 확인이 정상일 때만 노출
- 가입 직전 마지막 확인 단계
- 블록 자체는 짧고 압축적으로 보여야 한다.

### Step 7. 회원가입 버튼
- 마지막 단계에서만 화면 내 주 액션으로 강하게 등장
- 이전 단계에서는 버튼이 과도하게 시선을 끌지 않도록 한다.

## 정보 구조
1. 뒤로가기
2. 브랜드/설명 카피
3. 현재 보이는 입력
4. 조건 충족 후 다음 입력 reveal

## UI 구성 원칙
- 진행 상황 컴포넌트, 스텝 바, 요약 카드 사용 금지
- 큰 메인 카드 사용 금지
- 카드 안에 카드가 반복되는 구조 금지
- 구분은 배경 박스가 아니라 여백, 얇은 선, 타이포, 상태 문구로 해결
- 입력 하나하나는 또렷하되, 화면 전체는 가볍게 느껴져야 한다.

## 상태 설계
### 기본 상태
- 현재 입력 대기

### 노출 상태
- 이전 조건 충족 후 다음 입력 등장

### 오류 상태
- 이메일 형식 오류
- 비밀번호 불일치
- 인증 미완료 상태
- 약관 미동의 상태

### 원칙
- 상태는 입력 바로 주변에서 해결한다.
- 별도의 진행 현황 UI로 상태를 설명하지 않는다.

## 카피 가이드
- 짧고 직접적인 안내 문장 사용
- "다음 단계를 진행하세요" 같은 메타 문구보다 입력 자체가 흐름을 설명하게 한다
- 에러는 차갑지 않되 명확하게
- 예:
  - `올바른 이메일 형식을 입력해주세요`
  - `비밀번호가 일치하지 않습니다`
  - `이메일 인증을 완료해주세요`

## 비주얼 가이드
- 현재 다크 배경과 teal 포인트는 유지
- 카드형 폼 대신 배경과 입력이 자연스럽게 이어지는 레이아웃 사용
- 입력 영역은 충분히 넓어 보여야 하며, 특히 모바일에서 박스 안에 갇힌 느낌이 없어야 한다
- 배경 효과는 유지 가능하지만 입력 가독성을 방해하면 안 된다
- 여백은 `넓고 시원하게`, 박스는 `최소한으로` 가져간다

## 레이아웃 가이드
### 데스크톱
- 좌측에는 짧은 브랜드 카피
- 우측 또는 중앙축에는 입력 흐름
- 단, 입력 영역도 카드처럼 뜨지 말고 배경 위에 직접 놓인 인상이어야 한다.

### 모바일
- 단일 컬럼
- 입력 폭을 최대한 넓게 확보
- 큰 바깥 카드 없이 화면과 입력이 자연스럽게 붙어 있는 인상 유지
- 세로 리듬을 이용해 다음 입력이 아래로 이어지게 한다.

## 모션 가이드
- 다음 입력이 열릴 때 아래에서 위로 짧게 등장
- 오류는 흔들림보다 색상과 짧은 문구 중심
- 전체 모션은 적고 절제되어야 한다
- 입력 reveal 자체가 핵심 인터랙션이어야 한다

## 권장 레이아웃 패턴
### 옵션 A. 화면 일체형 단일 컬럼 reveal
- 가장 추천하는 구조
- 화면과 입력이 하나의 레이어처럼 이어진다
- 모바일 대응이 가장 자연스럽다

### 옵션 B. 좌측 카피 + 우측 입력 흐름
- 데스크톱에서만 제한적으로 사용 가능
- 모바일에서는 반드시 단일 컬럼으로 접혀야 한다

이번 요구사항에서는 옵션 A를 기본안으로 채택한다.
